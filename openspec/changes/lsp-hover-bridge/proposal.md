# Proposal: LSP Bridge & Hover Tool

## Problem Statement
When the agent works with external dependencies (like React, Axum, or Serde), `viking-context` correctly avoids indexing them to save VRAM and crawling time. However, this leaves the Tier 2 LLM blind to the exact signatures and generic constraints of external functions, leading to hallucinations or compilation errors.

## Vision
We will leverage the developer's IDE as a semantic oracle. Since VS Code already runs a Language Server (`rust-analyzer`, `tsserver`, etc.), we will add an `AskLsp` tool to the `faas/orchestrator`. 
When the agent needs to understand an external function call, it will ask the orchestrator to trigger a virtual "Mouse Hover" at specific coordinates in the codebase. The `pulsar-vscode` extension will execute this native VS Code command and return the official Markdown documentation and signature generated by the LSP.

## Value Proposition
- **Zero Hallucination:** The agent gets compiler-grade truth for API signatures.
- **VRAM & Token Economy:** No need to read raw library files; the agent only requests the specific signature it needs.
- **100% Offline & Sovereign:** No web searches required. The entire pipeline relies on the local IDE.
